# 销售管理系统开发执行文档

版本：v0.1  
生成日期：2026-04-29  
文档定位：给后续开发 agents 使用的产品、架构、接口、数据、测试与协作执行规格。  
原始需求来源：[需求文档.md](../需求文档.md)

## 1. 决策摘要

本项目定位为面向中小团队的轻量级 Web 销售管理系统，不做电商前台，不直接复制完整 ERP/CRM，而是先完成“商品、客户、订单、库存、收款、报表、权限、日志、Docker 交付”的内部管理闭环。

首版技术栈建议：

| 层级 | 选型 | 原因 |
|---|---|---|
| 前端 | React + TypeScript + Vite | 管理后台适合组件化表格/表单/图表；Vite 支持 `react-ts` 模板，启动和构建快 |
| UI 组件 | Ant Design | 表格、表单、弹窗、权限菜单、后台布局成熟 |
| 后端 | FastAPI | API-first、自动 OpenAPI 文档、适合前后端分离和多 agent 并行 |
| ORM/迁移 | SQLAlchemy 2.x + Alembic | 关系模型、事务、迁移能力成熟 |
| 数据库 | PostgreSQL | 适合订单、库存、报表、审计等关系型业务 |
| 缓存 | Redis，可选 | 首版可不用；后续用于限流、报表缓存、异步任务 |
| 部署 | Docker Compose | 满足 Docker 开发、Windows 兼容和一键启动要求 |

关键产品决策：

- MVP 必须包含客户管理，否则销售订单无法形成真实业务闭环。
- 商品录入流程必须足够简单：前端首版只强制输入商品名称、商品图片、成本价、销售价，其余字段由系统自动生成或使用默认值。
- 商品图片纳入 MVP：后端负责上传、存储、访问 URL、删除清理和文件校验，前端只提供选择/拖拽上传与预览。
- 商品改价不能影响历史订单，订单明细必须保存商品名称、SKU、成交价、成本价快照。
- 已被订单引用的商品、客户、用户默认禁止硬删除，使用停用或软删除。
- 金额计算统一在后端完成，使用 Decimal 或数据库 `numeric`，禁止用浮点数处理金额。
- 利润、毛利率、折扣金额、折扣率、销售排行、利润排行、默认排序全部由后端计算，前端不手工维护这些派生字段。
- 权限不能只靠前端隐藏菜单，后端每个业务接口都必须校验身份、角色和数据范围。
- 每个开发阶段结束必须更新文档、测试报告和交接记录。

## 3. 产品范围

### 3.1 产品目标

系统要解决以下问题：

1. 统一维护销售商品、价格、成本、库存状态。
2. 统一维护客户资料、联系人、归属销售和跟进状态。
3. 支持销售人员创建订单，系统自动计算销售额、成本、毛利和应收。
4. 支持库存扣减、库存调整和库存流水追溯。
5. 支持财务登记收款，管理未收、部分收、已收状态。
6. 为老板、主管、财务提供销售额、毛利、商品排行、客户排行等报表。
7. 用角色权限隔离不同岗位的数据访问和操作能力。
8. 通过 Docker 保证开发、测试、部署环境一致。
9. 通过 `.agent_workspace/` 保证多个 agents 可以连续接手开发。

### 3.2 非目标

MVP 不实现以下内容：

- 电商前台、在线购物车、外部支付。
- 复杂销售漏斗自动化和营销活动。
- 电子合同、电子签、发票系统。
- 多仓库、采购补货、供应商管理。
- 复杂退换货、部分退款、财务对账。
- 移动端 App。
- AI 预测、自动推荐、语音/客服集成。

这些能力进入后续 backlog，不得阻塞 MVP。

### 3.3 MVP 验收闭环

一个销售人员可以完成：

1. 登录系统。
2. 新增商品，只输入商品名称、图片、成本价、销售价，系统自动生成 SKU、默认分类、默认状态、默认库存和利润指标。
3. 新增客户，记录联系人。
4. 创建销售订单，选择客户和商品。
5. 系统计算订单金额、成本、毛利。
6. 确认订单时扣减库存。
7. 财务登记收款。
8. 管理者查看销售额、毛利、商品排行和订单列表。
9. 管理员查看关键操作日志。

只要这条链路不能跑通，就不能判定 MVP 完成。

## 4. 用户角色与权限

| 角色 | 目标 | 数据范围 | 关键权限 |
|---|---|---|---|
| 系统管理员 | 配置系统、用户、角色、基础数据 | 全部 | 用户/角色/权限、全部业务数据、导出、审计 |
| 销售人员 | 维护客户、创建订单、跟进成交 | 本人客户、本人订单 | 客户 CRUD、订单创建、本人业绩 |
| 销售主管 | 管理团队过程和业绩 | 团队客户、团队订单 | 团队数据、低毛利/大折扣审批、客户分配 |
| 库存/运营 | 维护商品和库存、处理交付 | 商品、库存、待交付订单 | 商品维护、库存调整、交付处理、库存流水 |
| 财务/老板 | 查看金额、利润、应收，登记收款 | 全部或授权范围 | 收款、应收、利润报表、导出 |
| 只读审计 | 查看日志和指定数据 | 授权范围 | 只读查看、审计日志 |

敏感字段：

- `cost_price` 成本价
- `gross_profit` 毛利
- `gross_margin` 毛利率
- 全局利润报表

默认仅系统管理员、销售主管、财务/老板可见。其他角色需要显式授权。

权限设计必须支持：

- 菜单权限
- API 操作权限
- 数据范围权限：本人、团队、全部
- 敏感字段可见权限

## 5. 核心业务流程

### 5.1 商品建档流程

1. 运营或管理员点击“新增商品”。
2. 前端只要求填写商品名称、上传商品主图、填写成本价和销售价。
3. 后端自动生成 SKU、默认分类、默认状态、默认库存、排序权重和利润指标。
4. 后端校验商品名称、图片格式、金额非负、销售价和成本价精度。
5. 保存商品、图片元数据和价格/成本历史，并写入操作日志。
6. 商品状态默认为“上架”，可直接被销售订单选择。

自动处理规则：

| 项目 | 自动规则 |
|---|---|
| SKU | 按 `SPU-YYYYMMDD-四位序号` 生成，保证唯一；后续可允许管理员手工修改 |
| 分类 | 默认进入“未分类”；后续可由管理员调整 |
| 状态 | 默认上架 |
| 库存 | 默认 0；如果业务希望弱化库存，订单确认时可配置为不校验库存 |
| 利润 | `sale_price - cost_price` |
| 毛利率 | `利润 / sale_price`，销售价为 0 时返回 0 |
| 默认排序 | 新商品优先，其次按手工置顶、销量、更新时间综合排序 |
| 销售排行 | 根据已确认/已完成订单中的销售数量和销售额自动聚合 |
| 利润排行 | 根据订单明细快照成本和成交价自动聚合 |

异常规则：

- SKU 重复：拒绝保存。
- 价格或成本为负：拒绝保存。
- 图片格式不是 jpg、jpeg、png、webp：拒绝上传。
- 图片超过配置大小：拒绝上传，默认单张不超过 5MB。
- 商品已有订单引用：禁止硬删除，只能停用。
- 成本价字段无权限用户不可见。

### 5.2 客户建档流程

1. 销售新增客户。
2. 录入客户名称、联系人、电话、来源、等级、归属销售。
3. 系统按手机号、客户名称进行重复提醒。
4. 销售可维护跟进状态和备注。
5. 主管可调整客户归属。

异常规则：

- 手机号格式错误：拒绝保存或提示格式异常。
- 客户重复：MVP 只提醒，不强制合并；后续版本支持合并。
- 销售人员不能查看其他销售客户，除非数据范围授权。

### 5.3 订单创建与确认流程

1. 销售创建草稿订单。
2. 选择客户和商品，填写数量、成交单价、折扣、备注。
3. 后端按订单明细计算总金额、总成本、毛利、毛利率。
4. 商品价格、成本、SKU、名称写入订单明细快照。
5. 销售提交确认。
6. 后端校验库存、商品状态、订单状态。
7. 若库存足够且不触发审批，订单变为已确认并扣减库存。
8. 若库存不足，订单保持草稿或进入待处理，前端给出明确提示。
9. 若低于成本价或折扣超过阈值，进入待审批。

MVP 可简化为：

```text
草稿 -> 已确认 -> 部分收款 -> 已完成
草稿 -> 已取消
已确认 -> 已取消，取消时回滚库存
```

V1.0 目标状态可扩展为：

```text
草稿 -> 待审批 -> 已确认 -> 待交付 -> 部分交付 -> 已交付 -> 部分收款 -> 已完成
草稿/待审批/已确认 -> 已取消
待审批 -> 已驳回
已完成 -> 已退款/部分退款，后续版本实现
```

关键约束：

- 已确认订单不能直接修改明细；需要取消后重建或后续实现调整单。
- 已完成订单不能删除。
- 订单状态流转必须由后端状态机控制。
- 库存扣减和订单确认必须在同一个数据库事务中完成。

### 5.4 收款流程

1. 财务进入订单详情。
2. 新增收款记录：金额、方式、时间、备注。
3. 后端校验收款金额不能超过剩余应收。
4. 更新订单已收金额。
5. 已收金额为 0：未收款。
6. 已收金额小于应收：部分收款。
7. 已收金额等于应收：已完成。
8. 写入收款记录和操作日志。

### 5.5 库存流程

库存变动来源：

- 商品初始库存
- 手工库存调整
- 订单确认扣减
- 订单取消回滚
- 后续版本的退货入库

库存流水必须记录：

- 商品
- 变动前数量
- 变动数量
- 变动后数量
- 变动类型
- 关联单据
- 操作人
- 操作时间

库存不能扣成负数。并发确认订单时，后端必须使用事务、行锁或乐观锁保护库存。

## 6. 功能模块规格

### 6.1 登录认证

MVP 功能：

- 用户名/密码登录。
- 退出登录。
- Access Token + Refresh Token。
- 当前用户信息接口。
- 登录失败提示统一，不暴露账号是否存在。

验收标准：

- 未登录访问业务接口返回 `AUTH_UNAUTHORIZED`。
- Token 过期后可通过 refresh 刷新。
- 禁用用户不能登录。
- 密码必须哈希存储，不得明文保存。

### 6.2 用户与权限

MVP 功能：

- 用户列表、新增、编辑、禁用。
- 角色列表、新增、编辑。
- 角色绑定权限。
- 用户绑定角色。
- 默认初始化管理员账号。

验收标准：

- 普通销售无法访问用户管理。
- 销售只能看本人客户和订单。
- 主管能看团队数据。
- 敏感字段由后端控制是否返回。

### 6.3 商品管理

设计目标：前端录入尽可能轻，复杂字段由系统自动处理。新增商品首屏只展示 4 个必填项：商品名称、商品图片、成本价、销售价。高级字段折叠到“更多设置”，不影响快速录入。

字段：

| 字段 | 说明 | 校验 |
|---|---|---|
| image | 商品图片 | MVP 必填，支持 jpg/jpeg/png/webp，默认单张主图 |
| name | 商品名称 | 必填，长度 1-100 |
| cost_price | 成本价 | 必填，>= 0，敏感字段 |
| sale_price | 销售价 | 必填，>= 0 |
| SKU | 商品编码 | 自动生成，唯一；高级设置可编辑 |
| category_id | 分类 | 自动为“未分类”；高级设置可编辑 |
| stock_quantity | 库存数量 | 默认 0；高级设置可编辑，>= 0 整数 |
| status | 状态 | 默认上架；可切换上架、下架、停用 |
| sort_weight | 排序权重 | 默认 0；管理员可手工置顶或调整 |
| remark | 备注 | 可选，高级设置 |

派生字段由后端返回，前端只展示：

| 字段 | 计算方式 |
|---|---|
| `unit_profit` | `sale_price - cost_price` |
| `gross_margin` | `unit_profit / sale_price`，销售价为 0 时为 0 |
| `sales_quantity` | 已确认/已完成订单明细数量汇总 |
| `sales_amount` | 已确认/已完成订单明细销售额汇总 |
| `profit_amount` | 已确认/已完成订单明细毛利汇总 |
| `discount_amount` | 订单成交单价低于商品销售价时自动计算 |
| `discount_rate` | `discount_amount / sale_price`，销售价为 0 时为 0 |

页面能力：

- 商品列表：图片缩略图、名称、售价、成本、单件利润、毛利率、销量、销售额、状态；支持搜索、分类筛选、状态筛选、分页、排序。
- 新增商品：默认只展示名称、图片、成本价、销售价 4 个必填项。
- 编辑商品。
- 图片上传、预览、替换、删除。
- 停用商品。
- 删除商品：仅无引用时允许硬删除；否则提示改为停用。
- 查看价格/成本变更记录。
- 自动排序：默认按 `sort_weight desc, sales_quantity desc, updated_at desc`；用户可切换按销量、销售额、利润、毛利率、创建时间排序。

验收标准：

- 用户只填写商品名称、图片、成本价、销售价即可创建商品。
- 商品创建后自动生成 SKU、默认分类、状态、库存和利润指标。
- 图片上传后可在列表、详情、订单选择器中展示缩略图。
- SKU 重复时报错。
- 负数价格、负数库存无法保存。
- 已停用商品不能被新订单选择。
- 无成本权限用户看不到成本价和毛利。
- 销量、销售额、利润排行来自订单数据自动聚合，不允许前端手工填写。

页面能力：

- 订单列表：订单号、客户、销售、金额、毛利、状态、创建时间。
- 创建订单：客户选择、商品明细、数量、价格、折扣、库存提示。
- 订单详情：基础信息、明细、收款记录、库存流水、操作日志。
- 确认订单、取消订单、登记收款。

验收标准：

- 订单明细为空不能提交。
- 商品不存在、停用、下架不能下单。
- 库存不足不能确认，或进入明确的缺货状态。
- 修改商品价格后，历史订单金额不变化。
- 修改商品图片后，历史订单仍保留下单时的图片快照 URL。
- 订单中修改成交单价时，折扣金额和折扣率由后端自动计算。
- 订单状态不能非法跳转。

### 6.6 库存管理

MVP 功能：

- 商品当前库存查看。
- 库存手工调整。
- 订单确认扣减库存。
- 订单取消回滚库存。
- 库存流水查询。

验收标准：

- 库存变动必须产生流水。
- 库存不能小于 0。
- 库存调整需要权限。
- 库存流水能追溯关联订单。

### 6.7 收款管理

MVP 功能：

- 订单详情内新增收款记录。
- 收款列表。
- 应收、已收、未收统计。
- 收款方式：现金、转账、银行卡、其他。

验收标准：

- 收款金额不能为负。
- 收款金额不能超过剩余应收。
- 删除或冲正收款需要管理员或财务权限，并写入日志。

### 6.8 报表看板

MVP 指标：

- 今日销售额
- 本月销售额
- 订单数量
- 毛利
- 毛利率
- 热销商品排行
- 客户销售排行
- 销售人员业绩排行
- 库存预警列表

统计口径：

- 销售额：已确认、部分收款、已完成订单的 `total_amount`。
- 成本：同范围订单的 `total_cost`。
- 毛利：销售额 - 成本。
- 毛利率：毛利 / 销售额；销售额为 0 时返回 0。
- 订单数：按订单创建时间或确认时间筛选，默认按确认时间。
- 取消订单不计入销售额和毛利。

验收标准：

- 报表支持时间范围筛选。
- 空数据时显示空状态。
- 图表和表格数据一致。
- 无权限用户不能查看利润类指标。

### 6.9 操作日志与审计

必须记录：

- 登录成功/失败。
- 用户、角色、权限变更。
- 商品新增、编辑、停用、删除。
- 成本价变更。
- 客户新增、编辑、归属变更。
- 订单创建、确认、取消。
- 收款新增、冲正。
- 库存调整。

日志字段：

|---|---|

| action | 操作 |
| resource_type | 资源类型 |
| resource_id | 资源 ID |
| before_data | 变更前 |
| after_data | 变更后 |
| ip_address | IP |

### 8.2 图片上传与极简商品创建流程

推荐前端流程：

1. 用户在商品新增页填写商品名称、成本价、销售价。
2. 用户选择或拖拽商品图片。
3. 前端先调用 `POST /api/v1/files/images` 上传图片，获得 `file_id` 和 `url`。
4. 前端调用 `POST /api/v1/products`，只提交 `name`、`cost_price`、`sale_price`、`main_image_file_id`。
5. 后端自动生成 SKU、默认分类、默认状态、默认库存、排序权重、利润指标。
6. 前端跳转商品列表或商品详情。

商品创建最小请求：

```json
{
  "name": "示例商品",
  "main_image_file_id": "uuid",
  "cost_price": "12.50",
  "sale_price": "19.90"
}
```

商品创建最小响应：

```json
{
  "success": true,
  "data": {
    "id": "uuid",
    "sku": "SPU-20260429-0001",
    "name": "示例商品",
    "main_image_url": "/uploads/products/2026/04/uuid.webp",
    "cost_price": "12.50",
    "sale_price": "19.90",
    "unit_profit": "7.40",
    "gross_margin": "0.3719",
    "status": "active",
    "stock_quantity": 0
  },
  "message": "创建成功",
  "request_id": "req_xxx"
}
```

图片上传规则：

| 项目 | 规则 |
|---|---|
| 文件类型 | jpg、jpeg、png、webp |
| 文件大小 | 默认单张不超过 5MB，可通过环境变量配置 |
| 存储路径 | 本地开发默认 `backend/uploads/products/YYYY/MM/` |
| 文件名 | 后端生成 UUID 文件名，禁止直接使用用户原始文件名作为存储名 |
| 访问 URL | 后端返回相对 URL 或经 Nginx 暴露的静态访问地址 |
| 图片处理 | MVP 可保留原图；推荐同时生成缩略图，列表优先加载缩略图 |
| 安全校验 | 校验 MIME、扩展名、文件头；拒绝脚本伪装图片 |
| 清理策略 | 未绑定商品的临时图片超过 24 小时可清理 |

### 8.3 自动排序、利润和折扣规则

商品列表默认排序：

```text
sort_weight desc,
sales_quantity desc,
updated_at desc
```

可选排序：

| 排序项 | 后端字段或计算口径 |
|---|---|
| 最新创建 | `created_at desc` |
| 最近更新 | `updated_at desc` |
| 手工排序 | `sort_weight desc` |
| 销量最高 | 报表聚合 `sales_quantity desc` |
| 销售额最高 | 报表聚合 `sales_amount desc` |
| 利润最高 | 报表聚合 `profit_amount desc` |
| 毛利率最高 | `gross_margin desc` |

折扣规则：

- 商品销售价是默认标价。
- 创建订单时默认成交单价等于商品销售价。
- 销售人员修改成交单价后，后端自动计算折扣金额：`sale_price_snapshot - unit_price`。
- 折扣率：`discount_amount / sale_price_snapshot`。
- 成交单价低于成本价时，MVP 阻断提交；后续可改为进入主管审批。
- 折扣金额为负时视为加价销售，允许但必须清晰展示。

### 11.4 上传文件存储策略

MVP 使用本地文件存储，Docker 通过 volume 持久化：

```text
backend/uploads/
  products/
    2026/
      04/
        uuid.webp
        uuid_thumb.webp
```

要求：

- 数据库只保存文件元数据和访问路径，不把图片二进制存入 PostgreSQL。
- 本地开发、测试、生产都必须通过 volume 保留 `uploads`。
- Nginx 生产环境可直接托管 `/uploads/` 静态路径；后端仍负责权限校验和元数据管理。
- 删除商品时不立即物理删除图片，先解除引用；定时清理未引用文件。
- 备份策略必须同时包含 PostgreSQL 数据和 uploads 文件目录。
- 后续如果文件量增长，可平滑迁移到 MinIO 或 S3；`files.storage_type` 和 `object_key` 字段为此预留。

## 12. 安全与可靠性

必须实现：

- 密码使用 bcrypt 或 argon2 哈希。
- 登录接口增加失败次数限制，后续可用 Redis。
- CORS 生产环境禁止 `*`。
- 所有写接口校验权限。
- 所有按 ID 查询、修改、删除接口做对象级权限校验。
- 后端 schema 做输入校验，禁止直接信任前端。
- 图片上传必须校验文件类型、大小、扩展名和文件头，禁止把任意文件暴露为静态资源。
- API 返回不泄露堆栈和数据库错误。
- request_id 贯穿日志和响应。
- 敏感字段日志脱敏。

重点防范：

- 认证错误。
- 对象级越权。
- 功能级越权。
- 过度暴露字段。
- 批量赋值导致越权修改敏感字段。
- 生产配置错误。

## 13. 日志、备份与恢复

日志：

- 后端输出结构化 JSON 日志。
- Docker 环境输出到 stdout/stderr。
- 记录请求方法、路径、状态码、耗时、用户 ID、request_id。
- 密码、token、手机号、邮箱等脱敏。

备份：

- PostgreSQL 每日 `pg_dump`。
- 保留最近 7 天每日备份、最近 4 周周备份。
- 文件命名：`sales_mgmt_YYYYMMDD_HHMMSS.sql.gz`。
- 至少每个大版本做一次恢复演练。

恢复文档必须说明：

- 停止服务。
- 恢复数据库。
- 运行迁移。
- 启动服务。
- 验证健康检查和核心链路。

| 端到端测试 | 登录 -> 新增商品 -> 新增客户 -> 创建订单 -> 确认订单 -> 收款 -> 看报表 |
| 回归测试 | 每阶段结束和发布前执行 |

### 16.2 MVP 必测清单

商品：

- 只填写名称、图片、成本价、销售价即可创建商品。
- 图片格式不支持时上传失败。
- 图片超过大小限制时上传失败。
- 商品创建后列表展示图片缩略图。
- 商品创建后自动生成 SKU、默认状态和利润指标。
- 新增商品成功。
- 商品名称为空失败。
- SKU 重复失败。
- 销售价格为负失败。
- 成本价格为负失败。
- 库存为负失败。
- 编辑商品成功。
- 已有关联订单商品不能硬删除。
- 按销量、销售额、利润排序结果正确。

客户：

- 新增客户成功。
- 销售只能查看本人客户。
- 主管可查看团队客户。
- 重复手机号提示。

订单：

- 创建订单成功。
- 订单明细为空失败。
- 商品不存在失败。
- 商品停用不可下单。
- 库存不足确认失败。
- 订单金额、成本、毛利正确。
- 修改成交单价后折扣金额和折扣率正确。
- 商品改价后历史订单不变。
- 商品换图后历史订单图片快照不变。
- 取消订单回滚库存。
- 非法状态流转失败。

收款：

- 登记收款成功。
- 收款金额为负失败。
- 收款超过剩余应收失败。
- 部分收款状态正确。
- 全额收款状态正确。

权限：

- 未登录不可访问业务接口。
- 普通用户不可访问用户管理。
- 无权限用户不可见成本和利润。

报表：

- 无数据显示空状态。
- 时间范围筛选正确。
- 图表数据与表格数据一致。

## 17. 注释与文档规范

代码注释：

- 注释使用中文。
- 注释解释“为什么这样做”，不要重复代码表面含义。
- 金额计算、状态流转、权限判断、数据迁移必须加必要注释。
- 临时方案必须标明 TODO、阶段和后续处理。

推荐：

```text
// 使用订单创建时的商品快照价格，避免商品后续改价影响历史订单金额。
```

不推荐：

```text
// 设置价格
```

TODO 格式：

```text
// TODO(阶段4): 当前仅支持整单取消，后续补充部分退款能力。
```

每阶段结束必须更新：

- `README.md`
- `docs/api.md`
- `docs/database.md`
- `docs/testing.md`
- `.agent_workspace/changelog.md`
- `.agent_workspace/implementation-records/implemented-features.md`
- `.agent_workspace/issues/issue-register.md`
- `.agent_workspace/issues/recurring-issues.md`
- `.agent_workspace/test-reports/phase-xx-test-report.md`
- `.agent_workspace/handoff/active-session.md`

## 18. Definition of Done

功能只有同时满足以下条件，才可以标记完成：

- 功能符合本文档和任务验收标准。
- 前端、后端、数据库影响已同步完成。
- 必要的单元测试、接口测试或端到端测试已补充。
- Docker 开发环境可启动并验证该功能。
- 关键异常路径已处理。
- 用户可见文案为中文。
- 必要代码注释为中文。
- API 文档、数据文档、开发记录已更新。
- 已实现功能已写入 `.agent_workspace/implementation-records/implemented-features.md`。
- 当前任务遇到的问题已写入 `.agent_workspace/issues/issue-register.md`。
- 已检查 `.agent_workspace/issues/recurring-issues.md`，确认没有重复踩坑或已采取预防措施。
- `.agent_workspace/handoff/active-session.md` 已更新到可恢复状态，并写明下一步第一动作。
- 无明显控制台错误、服务启动错误、数据库迁移错误。
- 当前阶段测试报告已更新。

不得标记完成：

- 只有前端假数据，后端未接通。
- 只有 API，前端无入口且需求要求 Web 操作。
- 没有测试，也没有记录为什么暂不测试。
- 依赖手动改数据库但没有迁移脚本。
- 有未记录的临时方案。
- 功能已实现但没有实现记录。
- 中断前没有更新 `active-session.md`。
- 重复出现的问题没有登记到 `recurring-issues.md`。

## 19. 风险清单

| 风险 | 影响 | 应对 |
|---|---|---|
| 需求范围膨胀 | 周期失控 | 锁定 MVP，新增需求进 backlog |
| 接口口径不一致 | 联调成本高 | 先定义 API 文档和响应格式 |
| 金额精度错误 | 财务数据不可信 | Decimal/numeric，禁止 float |
| 商品改价影响历史订单 | 历史数据失真 | 订单明细保存快照 |
| 删除商品导致订单断链 | 无法追溯 | 禁止硬删除，使用停用 |
| Docker/Windows 兼容问题 | 开发环境不一致 | 统一 Compose，提供 Windows 文档 |
| 多 agent 修改冲突 | 覆盖工作 | 按模块拆分，交接记录写清楚 |
| 测试滞后 | 后期修复成本高 | 每阶段必须测试报告 |
| 权限模型后补 | 大量返工 | 阶段 2 先做权限骨架 |
| 统计口径不清 | 报表争议 | 本文档固定口径，变更写 ADR |
| 文档滞后 | 新 agent 接手困难 | DoD 强制文档更新 |
| 中断后无法恢复现场 | 重复摸索、误改文件 | 强制维护 `active-session.md` 和实现记录 |
| 同类问题反复出现 | 浪费时间、质量下降 | 建立 `issue-register.md` 和 `recurring-issues.md`，复发问题优先级提升 |
